## 0. 阅读前必读：三处重要勘误（与既有认知/文档冲突）

在执行任何改动前，必须先纠正三个认知偏差，否则会改错方向：

### 勘误 1：CLAUDE.md「只生成 AI 封面」铁律与代码不符
- CLAUDE.md 写明"只生成 AI 封面（`cover_ai.png`），不生成多风格模板封面"。
- 但 `core/agents/cover_designer.py:134` 实际调用的是 `generate_cover_template(...)`（模板大字报封面），只是文件名/变量名仍叫 `cover_ai.png` / `ai_result`。
- **结论**：main 分支当前产出的是**模板封面**，不是 AI 绘画封面。本计划中所有"封面"改造，针对的是模板封面，不是 AI 封面。
- **后续动作**：同步修订 CLAUDE.md，消除文档与代码的矛盾（见任务 D-04）。

### 勘误 2：本计划的封面诊断需基于 main 当前产出重新实测
- 诊断阶段我实测的封面图来自 `agent-redesign` worktree（旧分支、macOS 生成、纯 AI 绘画），**不代表 main 当前模板封面的真实效果**。
- 因此封面任务（P1）的设计细节**必须等"阶段 0 实测"产出当前封面后再定稿**，不能直接套用旧样本结论。
- 已确定的、与封面方案无关的硬改（如标签、选题）可立即推进。

### 勘误 3：标签规则"早已存在但没生效"——根因是执行层，不是规则层
- 规则其实齐全：`prompts/agent_writer.md:292` 要求 3-5 个、`prompts/agent_editor.md:75` 规定超量记 issue、`core/publish_helpers.py:80-84` 有数量校验。
- 但实际产出仍是 15 个泛标签 → 说明 **LLM 没遵守 prompt 规则，且审核 Agent 没把它当硬伤打回**。
- **修复原则**：凡是"prompt 写了但 LLM 不稳定遵守"的规则，一律加**确定性代码后处理/校验**兜底，不再依赖 LLM 自觉（见任务 D-01）。

## 1. 问题诊断摘要

### 1.1 核心结论（一句话）
30 篇 0 涨粉 0 评论，不是"内容写得不好"，而是**三道流量闸门全关**：封面进不来人（CTR 6.1%）→ 内页留不住人（停留 18s）→ 账号没 IP 让人想关注。其中第一道闸门的旧方案（AI 封面）已被代码废弃，当前模板封面效果待实测。

### 1.2 数据漏斗（来自 `data/performance.json`，30 篇汇总）

| 指标 | 现状 | 赛道及格线 | 漏斗环节 | 归因 |
|---|---|---|---|---|
| 曝光→点击 CTR | **6.1%** | 8%+ | 第一道闸 | 封面 |
| 平均停留 | **18.1s** | 35s+ | 第二道闸 | 内页 |
| 点击→互动率 | **<1%**（816 浏览→6 互动） | 3%+ | 第二道闸 | 内页+钩子 |
| 评论 | **0** | — | — | 钩子+内容双失灵 |
| 涨粉 | **0** | — | 第三道闸 | 无人设/无系列 |

### 1.3 一个反直觉的佐证
performance 中**系统外手写笔记**（"为什么把天聊死"134 浏览、"分手信号"164 浏览）数据普遍**好于 AI 生成笔记**（浏览多在 5-42）。算法与用户都在识别 AI 套路感。→ 改造方向应是"去套路化 + 增真实感"，而非"写得更精美"。

### D-01｜标签确定性兜底（P0，立即止血）

**现状**：
- `core/agents/emotional_writer.py:348` 与 `prompts/agent_writer.md:292` 都要求 3-5 个，但 LLM 实际产出 15 个泛标签。
- 发布侧 `tabs/publish.py:83-118` 只在 UI 上做"提醒"，不在生成环节强制。

**问题**：标签堆砌 + 全泛词（#情感 #恋爱）触发小红书营销号降权，每篇发布都在主动招降权。这是当前**确定性、可立即止住**的流血点。

**改法**：在 `emotional_writer` 产出 `【话题标签】` 之后、写入 `note.md` 之前，加一层确定性后处理函数（纯 Python，不依赖 LLM）：

```
新增函数（建议放 core/utils.py 或 emotional_writer.py）：
  sanitize_tags(raw_tag_line: str) -> str
    1. 解析出所有 #标签
    2. 泛词黑名单过滤：#情感 #恋爱 #两性关系 #爱情 #生活 #日常 #心理 #女生必看 #治愈 #思考 #反思
       （这些是营销号高频泛词，搜索价值≈0 且招降权）
    3. 必带词兜底：若无 "#不懂就问有问必答" 则补在首位
    4. 截断到最多 5 个；若不足 3 个，从 keywords 字段补精准长尾词
    5. 重新拼成 "标签1 标签2 ..."（空格分隔，无#重复）
```

**落点**：
- 函数实现：`core/utils.py`（已有 `extract_json_from_llm` 等，风格一致）。
- 调用点：`core/agents/emotional_writer.py` 写 note.md 前；同时 `pipeline.py` 兼容层写文件前。
- 配套：`core/publish_helpers.py:check_compliance` 已有 3-5 校验，复用其逻辑，保持单一数据源。

**验收**：
- 新增单测 `tests/test_utils.py::test_sanitize_tags`：输入 15 个含泛词的标签 → 输出 ≤5 个、无泛词、含必带词。
- 跑 1 篇生成，确认 `note.md` 标签段 ≤5 个且无泛词。

**风险**：泛词黑名单需人工复核（某些词可能对特定选题有搜索价值）。→ 黑名单写成可配置常量，便于调整。

### D-02｜标签审核硬挂钩（P0）

**现状**：`agent_editor.md:75` 把"标签超量"列为 issue，但 `ChiefEditor` 决策逻辑（`core/agents/chief_editor.py`）未必把它当"必须打回"的硬伤，导致审核形同虚设。

**改法**：在 `ChiefEditor` 决策入口加一条**确定性预检**（在 LLM 审核之前/之后均可，建议之前，省 token）：

```
预检规则（chief_editor.py）：
  解析 note.md 的【话题标签】段：
    - 标签数 > 5 或 < 3  →  verdict 强制 = "conditional"，追加 issue"标签数量不合规"
    - 含泛词黑名单词    →  追加 issue"含降权泛标签"
  此预检结果作为硬门槛，LLM 审核不得覆盖为 pass。
```

**落点**：`core/agents/chief_editor.py`（决策函数入口）。

**验收**：
- 单测：构造一篇 15 标签的 note → `ChiefEditor` 必然返回 conditional/fail，不会被 LLM 判 pass。
- 与 D-01 双保险：即使 LLM 仍乱写，生成层（D-01）和审核层（D-02）任一生效即可拦截。

**风险**：低。预检只做"不通过"的收紧，不改既有通过逻辑。

### D-03｜情绪光谱→配色强制映射（P1，待 D-00）

**现状**：
- `render.py:50-93` `PALETTE` 已定义 7 套配色，`crimson`（正红/愤怒）、`mist`（蓝灰/代理正义）都在，但几乎没被用。
- `cover_designer.py:14-25` 的 `_STYLE_WEIGHTS` 偏向暖调，且 `_COLD_STYLES`（twilight/cool/mist）有 60% 概率被强制替换成暖调（`cover_designer.py:109-115`）。→ 这是"视觉全暖、配色单一"的代码根因。
- 写作 prompt `agent_writer.md:91-102` 本来定义了"情绪基调→视觉风格映射"，但执行链没打通。

**问题**：情感号需要"颜色即情绪"。愤怒共鸣该用 crimson、代理正义该用 mist，但代码强行把它们换回暖灰。20 篇样本 warm_grey×12 + twilight×6 就是结果。

**改法**：
1. **删除/弱化冷调强制降权**：`cover_designer.py:109-115` 的 60% 替换逻辑，改为"仅当 LLM 无明确情绪理由时才降权"，或直接让 crimson/mist 与 warm_grey 平权。
2. **情绪→配色确定性映射**：从 note.md 的 `【视觉风格】` 字段读取（由写作 Agent 按情绪选定），`CoverDesigner` **尊重该字段**，不再随机覆盖。即把 `agent_writer.md` 已定义的映射变成"写作 Agent 定调 → 封面 Agent 执行"的闭环。
3. **保留风格轮换**：`_enforce_rotation`（防连续 3 次同风格）继续生效，但轮换池扩大到包含 crimson/mist。

**落点**：
- `core/agents/cover_designer.py:109-115`（冷调降权）、`:14-25`（权重）、`:101-107`（style_override 逻辑，已有雏形，强化为"以写作 Agent 定调为准"）。
- `prompts/agent_writer.md:91-102`：明确"必须先定情绪基调，再输出对应的视觉风格字段，不得全部写 warm_grey"。

**验收**：
- 连续生成 10 篇，统计 `【视觉风格】` 分布，crimson/mist/cool 合计占比 ≥30%（不再全暖）。
- 实测 crimson 封面在信息流中的对比度是否足够（待 D-04 一起验）。

**风险**：crimson 用不好会显廉价。→ 配合 D-00 实测调整 `PALETTE.crimson` 的 `cover_accent` 饱和度（`render.py:85`）。

### D-05｜内页聊天气泡渲染（P1，高价值）

**现状**：
- 写作 prompt `agent_writer.md:64-68`（结构 H）和大量案例都让写手产出"对话体"（"她：... 他：..."）。
- 但 `core/image/inner_page.py:_parse_to_blocks`（`:25-57`）把**所有文本当普通段落**，对话被拍扁成纯文字行。→ 写作策略设计的爆款结构，到制图层完全失效。这是当前最大的"策略-执行脱节"。

**问题**：内页纯素文字墙，无场景化视觉，停留 18s 的主因之一。

**改法**：扩展 block 类型，识别对话行并渲染成气泡：

```
1. _parse_to_blocks 增加 block 类型：
   - 普通段落 (type="text")
   - 对话行 (type="bubble", side="left"|"right")：
     识别规则：行首匹配 ^(她|他|我|妈妈|男生|女生|对方)[：:] → 取说话人侧
     （可约定：主角=右气泡暖色，对方=左气泡冷色，或按角色映射）
2. _paginate_blocks / generate_inner_page 渲染分支：
   - bubble 类型画圆角矩形气泡 + 可选说话人小标签
   - 左右交替排版，模拟聊天界面
3. 保留现有金句高亮、页码、装饰逻辑不变
```

**落点**：`core/image/inner_page.py`（解析 `:25-57` + 渲染 `:209-334`）。

**验收**：
- 单测：输入含"她：今天好累\n他：早点睡"的正文 → 生成图含两个左右气泡。
- 实测：D-00 那篇笔记的内页，对话段以气泡呈现，不再是纯文字。
- 停留时长待发布后回看（目标 18s → 30s+）。

**风险**：气泡排版可能挤占竖向空间，导致页数增多。→ 分页逻辑 `_paginate_blocks` 需同步给气泡算高度；必要时允许单页气泡数上限。

### D-06｜内页金句强化（P1，小改动高收益）

**现状**：`inner_page.py:241-264` 已有"最后一页最后一段金句放大"逻辑，但触发条件苛刻（`is_last_page and is_last_text_block and is_short`），且 `agent_writer.md` 要求"金句单独成页像海报"未在渲染层保证。

3. 金句页加更醒目的高亮底/边框（复用 `:266-292` 的色块逻辑，加粗）。

**落点**：`core/image/inner_page.py` + `core/image/render.py:31-47`（LAYOUT 常量）。

**验收**：每篇笔记至少有 1 页是"纯金句海报页"，可被截图传播。

**风险**：低。纯调参与触发条件放宽。

### D-07｜选题方向拓宽（P1）

**现状**：
- `generate_topics.py` 生成器完善（有故事原型、争议锚点、热点融合），但 `SYSTEM_PROMPT`（`:16-31`）和 `USER_PROMPT`（`:33-80`）的 5 个 pillar 与情绪方向偏窄。
- 现有 50 选题高度集中在"关系里的隐性伤害"（情绪吸血鬼/边界感/讨好型/为你好…），单一情绪带，且缺搜索型选题。

**问题**：赛道自我锁死，起不来量；缺搜索词，拿不到长尾流量。

**改法**：
1. **改 `generate_topics.py` 的 prompt**：
   - 在 `SYSTEM_PROMPT` 增加"情绪光谱均衡"约束：每批必须覆盖【愤怒共鸣/搞笑自嘲/怀旧刺痛/疗愈陪伴/实用洞察】至少 4 种，禁止全"丧/清醒/止损"。
   - 在 `USER_PROMPT` 增加"搜索型选题配额"：每批至少 2 个是用户会主动搜索的问句（如"30岁不结婚真的可怕吗""怎么判断男朋友爱不爱你"），带 `keywords` 用于 SEO。
2. **存量选题治理**：对 `data/topics.json` 现有 50 个做一次人工/AI 标注，标记"隐性伤害"类占比，砍掉冗余的 1/3，腾出位置生成新方向。

**落点**：`generate_topics.py:16-80`（prompt）+ `data/topics.json`（治理）。

**验收**：
